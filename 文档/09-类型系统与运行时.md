# 9. 类型系统与运行时模型

"道"选择动态类型系统和基于字节码的运行时，旨在平衡开发效率、灵活性和跨平台性能。

## 9.1 动态类型系统

### 9.1.1 优势与挑战

动态类型系统允许变量在运行时持有不同类型的值，这极大地提高了编程的灵活性和表达力，并减少了样板代码。然而，这也带来了一些挑战，最主要的是许多类型错误只有在运行时才能被发现，以及动态派发可能带来的性能开销。

### 9.1.2 增强方案：可选类型注解与检查

为了弥补动态类型的不足，特别是在大型项目中，"道"生态系统将提供一个独立的静态分析工具（类似 Python 的 mypy 或 JavaScript 的 TypeScript）。

**类型注解语法：**

```
// 基本类型注解
函数 打招呼(名字: 文本) -> 文本
    返回 `你好, {名字}`

// 集合类型注解
函数 求和(数字们: 列表[数值]) -> 数值
    返回 折叠(数字们, 0, 函数(a, b) => a + b)

// 可选类型（可能为空）
函数 查找用户(ID: 数值) -> 用户 | 空
    // ...

// 联合类型
函数 处理输入(输入: 文本 | 数值) -> 文本
    匹配 输入
        情况 值: 文本: 返回 值
        情况 值: 数值: 返回 `{值}`

// 字典类型注解
函数 处理配置(配置: 字典[文本, 任意]) -> 布尔
    // ...
```

**类型推断：** 该工具将采用先进的类型推断算法（如笛卡尔积算法 CPA 的变体）来推断未注解代码的类型。它可以分析源代码甚至编译后的字节码来推断类型信息。

**类型别名：**

```
// 定义类型别名，简化复杂类型
类型别名 用户ID = 数值
类型别名 用户列表 = 列表[用户]
类型别名 处理器 = 函数(请求) -> 响应

函数 获取用户(id: 用户ID) -> 用户 | 空
    // ...
```

## 9.2 编译为中间字节码

将"道"源代码编译为中间字节码是实现平台无关性和高性能的关键。

### 9.2.1 模型优势

* **跨平台性**：同一份字节码可以在任何安装了"道虚拟机"（DVM）的平台上运行。
* **性能优化**：虚拟机可以在运行时对字节码进行即时编译（JIT），将热点代码编译成本地机器码。
* **安全性**：代码运行在虚拟机的沙箱中，可以对其资源访问进行严格控制。
* **简化编译器**：编译器只需面向单一的字节码规范。

### 9.2.2 编译流程

```
源代码(.道文件)
    ↓ [词法分析]
Token 流
    ↓ [语法分析]
抽象语法树 (AST)
    ↓ [宏展开] ← 此阶段执行所有宏，生成新AST
展开后的 AST
    ↓ [语义分析 + 可选类型检查]
带注解的 AST
    ↓ [字节码生成]
字节码(.道码 文件)
    ↓ [虚拟机执行 / JIT编译]
运行结果
```

### 9.2.3 目标虚拟机平台分析

选择一个成熟、高性能的虚拟机作为"道"的执行后端至关重要。

| 平台 | 优势 | 挑战 | 适合度 |
|------|------|------|--------|
| **GraalVM** | 多语言设计；Truffle框架大幅简化实现；自动提供顶级JIT；零开销语言互操作 | 相对较新；学习曲线陡峭 | ⭐⭐⭐⭐⭐ |
| **JVM** | 极其成熟稳定；`invokedynamic` 支持动态语言；庞大生态 | 原为静态类型设计；启动慢、内存大 | ⭐⭐⭐⭐ |
| **.NET CLR** | 多语言支持；DLR 优秀；性能好 | 非 Windows 生态偏弱 | ⭐⭐⭐⭐ |
| **WASM** | 轻量、快速、安全；从浏览器到服务器 | GC/异常处理仍在发展；动态语言支持不成熟 | ⭐⭐⭐ |

**结论与推荐**：**GraalVM 是"道"最理想的虚拟机平台。** 它的多语言设计哲学与"道"的多范式目标完美契合。使用 Truffle 框架可以大大降低开发一个高性能 JIT 编译器的难度，并能天然地解决与其他主流语言的互操作问题。

### 9.2.4 集成挑战与解决方案

| 挑战 | 说明 | 解决方案 |
|------|------|---------|
| **宏展开** | 宏系统操作 AST，需在编译前端完成 | 在语法分析后、字节码生成前插入宏展开阶段 |
| **逻辑编程的回溯** | 标准 VM 的调用栈不支持回溯 | 使用独立的 trail stack 或 CPS 转换 |
| **动态类型与JIT** | 需要运行时类型信息才能生成高效机器码 | 利用内联缓存、类型反馈和推测优化 |
| **统一错误处理** | 跨范式的错误必须能被统一捕获 | 所有范式的错误统一转换为"道"的异常类型 |
