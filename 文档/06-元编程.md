# 6. 元编程/宏系统 (Metaprogramming/Macro System)

元编程是"代码写代码"的艺术。一个强大的宏系统能让开发者从根本上扩展语言的语法，创建领域特定语言（DSL），并消除重复的模板代码。"道"的宏系统以 Lisp 的同象性（Homoiconicity）思想为指导，并借鉴 Rust 宏的健壮性。

## 6.1 设计理念：受控的同象性

纯粹的同象性，即代码和数据具有完全相同的结构（如 Lisp 的 S 表达式），对于一种具有更传统语法的语言来说难以实现。我们将采用一种"受控的同象性"方法：

1. **代码即数据结构**：提供一种机制，可以将任意"道"代码块转换为语言内置的数据结构，即抽象语法树（AST）的表示。这个 AST 将由列表、字典和基本类型构成。
2. **宏在 AST 上操作**：宏本质上是特殊的函数，它们在编译期执行，接受 AST 作为输入，并返回一个新的 AST。
3. **语法安全**：宏系统应是卫生的（Hygienic Macros），以避免宏引入的变量与使用处的变量发生意外冲突。

## 6.2 核心语法

| 关键字 | 说明 |
|--------|------|
| `定义宏` | 定义一个编译期宏 |
| `引述` | 将代码块转换为 AST 数据（类似 Lisp 的 `quote`） |
| `注入` | 在引述块中插入变量的 AST 值（类似 Lisp 的 `unquote`） |
| `!宏名` | 宏调用前缀，区分宏与普通函数 |
| `匹配` | 在宏定义中标记模式匹配宏，后跟模式分支列表 |
| `=>` | 模式分支分隔符，左侧为匹配模式，右侧为引述体 |
| `当` | 守卫条件关键字，在模式后添加额外条件 |
| `$变量名` | `注入 变量名` 的语法糖 |

## 6.3 基础示例：除非宏

```
// 设计一个 `除非` 宏，作为 `如果` 的反义词
定义宏 除非(条件, 代码块)
    返回 引述
        如果 不是 (注入 条件)
            注入 代码块

// 使用宏
定义 温度 = 15
!除非(温度 > 20, {
    打印("天气有点冷，请加衣服。")
})

// 上述宏在编译期会被展开为：
// 如果 不是 (温度 > 20)
//     打印("天气有点冷，请加衣服。")
```

## 6.4 实用宏示例

### 6.4.1 计时宏

自动为任意代码块添加计时功能：

```
定义宏 计时(代码块)
    返回 引述
        定义 __开始 = 系统.当前时间()
        定义 __结果 = { 注入 代码块 }
        定义 __耗时 = 系统.当前时间() - __开始
        打印(`执行耗时: {__耗时} 毫秒`)
        __结果

// 使用
!计时({
    定义 总和 = 折叠(范围(1, 1000000), 0, 函数(a, b) => a + b)
    打印(`总和: {总和}`)
})
// 输出：
// 总和: 499999500000
// 执行耗时: 42 毫秒
```

### 6.4.2 重试宏

自动为可能失败的操作添加重试逻辑：

```
定义宏 重试(次数, 代码块)
    返回 引述
        定义 __剩余次数 = 注入 次数
        定义 __最后错误 = 空
        当 __剩余次数 > 0
            尝试
                { 注入 代码块 }
                跳出  // 成功则退出循环
            捕获 错误
                __最后错误 = 错误
                __剩余次数 = __剩余次数 - 1
                如果 __剩余次数 > 0
                    打印(`操作失败，剩余重试次数: {__剩余次数}`)
        如果 __最后错误 != 空 并且 __剩余次数 == 0
            抛出 __最后错误

// 使用：自动重试3次网络请求
!重试(3, {
    定义 响应 = 网络.请求("https://api.example.com/data")
    打印(响应.数据)
})
```

### 6.4.3 日志宏

自动记录函数的输入输出：

```
定义宏 记录日志(函数定义)
    // 解析函数定义的AST，提取函数名和参数
    定义 函数名 = 函数定义.名称
    定义 参数列表 = 函数定义.参数
    定义 函数体 = 函数定义.主体

    返回 引述
        函数 (注入 函数名)((注入 参数列表))
            打印(`[日志] 调用 {(注入 函数名)} 参数: {(注入 参数列表)}`)
            定义 __返回值 = { 注入 函数体 }
            打印(`[日志] 返回: {__返回值}`)
            返回 __返回值

// 使用
!记录日志(
    函数 计算价格(单价, 数量)
        返回 单价 * 数量
)

计算价格(100, 5)
// 输出：
// [日志] 调用 计算价格 参数: [100, 5]
// [日志] 返回: 500
```

### 6.4.4 构建器宏

为领域特定语言（DSL）提供便捷语法：

```
定义宏 网页(代码块)
    // 宏将DSL代码转换为HTML字符串
    返回 解析网页DSL(代码块)

// 使用DSL语法生成HTML
定义 页面 = !网页({
    标题 "我的主页"
    段落 "欢迎来到道语言的世界"
    列表
        项 "简洁"
        项 "强大"
        项 "中文原生"
})

// 展开后等效于：
// 定义 页面 = "<h1>我的主页</h1><p>欢迎来到道语言的世界</p><ul><li>简洁</li><li>强大</li><li>中文原生</li></ul>"
```

## 6.5 AST 内省

开发者可以在运行时检查代码的 AST 结构，用于调试和高级元编程：

```
定义 代码树 = 引述
    函数 你好(名字)
        打印(`你好, {名字}`)

// 检查AST结构
打印(代码树.类型)     // "函数定义"
打印(代码树.名称)     // "你好"
打印(代码树.参数)     // ["名字"]
打印(代码树.子节点)   // [打印调用节点...]
```

## 6.6 宏的卫生性

"道"的宏系统默认是卫生的，宏内部定义的变量不会与调用处的变量冲突：

```
定义宏 交换(甲, 乙)
    返回 引述
        定义 临时 = 注入 甲     // 这里的 `临时` 不会与调用者的同名变量冲突
        注入 甲 = 注入 乙
        注入 乙 = 临时

定义 临时 = "不会被影响"
定义 x = 1
定义 y = 2
!交换(x, y)
打印(x)   // 2
打印(y)   // 1
打印(临时) // "不会被影响" （宏内部的 `临时` 与此处的 `临时` 是不同的变量）
```

## 6.7 模式匹配宏

模式匹配宏是普通宏的扩展形式，允许根据参数的结构进行分支展开。它借鉴了 Rust 的 `macro_rules!` 和 Elixir 的函数头模式匹配思想，将模式匹配与宏系统结合，实现编译期的模式匹配代码生成。

### 6.7.1 设计动机

普通宏通过参数替换进行展开，所有参数被统一处理。但在许多场景下，我们希望根据参数的**值或结构**选择不同的展开方式：

- 根据字面量值分派（如 `0` 和 `1` 生成不同的代码）
- 根据数据结构解构（如从列表中提取头部元素）
- 根据类型分派（如对列表和数值执行不同的逻辑）

模式匹配宏让这些场景可以用声明式的方式表达，而非在宏体中编写条件判断。

### 6.7.2 语法

```
定义宏 名称(参数) 匹配
    模式1 [当 守卫条件] => 引述体1
    模式2 [当 守卫条件] => 引述体2
    ...
```

与普通宏的区别在于：参数列表后紧跟 `匹配` 关键字，宏体由多个 `模式 => 引述体` 分支组成，而非单一的引述块。

### 6.7.3 匹配模式类型

| 模式类型 | 语法 | 说明 |
|----------|------|------|
| 字面量模式 | `0`、`"hello"`、`真`、`空` | 匹配特定值 |
| 变量绑定模式 | `x`、`名字` | 匹配任意值并绑定到变量 |
| 通配符模式 | `_` | 匹配任意值但不绑定 |
| 列表解构模式 | `[x, y]`、`[头, ...尾]` | 匹配列表并解构元素 |
| 字典解构模式 | `{"姓名": 名字}` | 匹配字典并解构键值对 |
| 类型检查模式 | `类型:列表` | 匹配特定类型的值 |
| 枚举变体模式 | `结果.成功(值)` | 匹配枚举类型的特定变体 |
| 守卫条件 | `x 当 x > 0` | 在模式匹配基础上添加额外条件 |

### 6.7.4 基础示例

#### 数值描述宏

```
定义宏 描述(值) 匹配
    0 => 引述 "零"
    1 => 引述 "一"
    _ => 引述 "其他"

设 结果1 = !描述(0)    // "零"
设 结果2 = !描述(1)    // "一"
设 结果3 = !描述(42)   // "其他"
```

模式按定义顺序从上到下匹配，第一个匹配成功的分支生效。通配符 `_` 通常作为最后一个分支，用于匹配所有未被子模式捕获的值。

#### 变量绑定

```
定义宏 传递(值) 匹配
    x => 引述 $x

设 结果 = !传递(42)   // 42
```

变量绑定模式 `x` 匹配任意值，并将值绑定到变量 `x`。在引述体中通过 `$x` 引用绑定的值。

### 6.7.5 列表解构

```
定义宏 头部(列表) 匹配
    [头, ...尾] => 引述 $头

设 结果 = !头部([10, 20, 30])   // 10
```

- `[头, ...尾]`：匹配至少包含一个元素的列表，将第一个元素绑定到 `头`，剩余元素绑定到 `尾`
- `[x, y]`：匹配恰好包含两个元素的列表
- `[]`：匹配空列表

### 6.7.6 字典解构

```
定义宏 取名(数据) 匹配
    {"姓名": 名字} => 引述 $名字

设 结果 = !取名({"姓名": "张三", "年龄": 25})   // "张三"
```

字典解构模式仅匹配字典中包含的键，不要求完全匹配所有键。`{}` 匹配任意字典。

### 6.7.7 类型检查模式

```
定义宏 类型描述(值) 匹配
    类型:列表 => 引述 "是列表"
    类型:数值 => 引述 "是数值"
    类型:文本 => 引述 "是文本"
    _ => 引述 "未知类型"

设 类型1 = !类型描述([1, 2, 3])   // "是列表"
设 类型2 = !类型描述(42)          // "是数值"
设 类型3 = !类型描述("你好")      // "是文本"
```

支持的类型名：`列表`、`字典`、`数值`、`文本`、`布尔`、`函数`。

### 6.7.8 枚举变体模式

```
枚举 结果
    成功
    失败

定义宏 处理结果(值) 匹配
    结果.成功(数据) => 引述 $数据
    结果.失败(信息) => 引述 $信息
```

枚举变体模式用于对代数数据类型（ADT）进行模式匹配，提取变体内部绑定的值。

### 6.7.9 守卫条件

守卫条件允许在模式匹配成功后添加额外的条件判断：

```
定义宏 分类(值) 匹配
    x 当 x > 0 => 引述 "正数"
    x 当 x < 0 => 引述 "负数"
    _ => 引述 "零"

设 分类1 = !分类(5)    // "正数"
设 分类2 = !分类(-3)   // "负数"
设 分类3 = !分类(0)    // "零"
```

模式匹配成功后，还需满足守卫条件（`当` 后的布尔表达式）才能进入该分支。守卫为假时，跳过当前分支，继续匹配下一个分支。

### 6.7.10 嵌套模式

模式支持嵌套，列表和字典模式内部可包含其他模式：

```
定义宏 检查(值) 匹配
    [0, y] => 引述 $y
    {"键": [x, y]} => 引述 $x + $y
```

### 6.7.11 与普通宏的兼容

模式匹配宏与普通宏共享同一套机制：

1. **统一注册表**：两种宏存储在同一个宏注册表中，不可同名
2. **统一调用语法**：都使用 `!宏名(参数)` 调用
3. **自动分派**：宏展开器根据宏定义类型自动选择展开方式

```
// 普通宏和模式匹配宏可以共存
定义宏 加一(x)
    返回 引述 $x + 1

定义宏 描述(值) 匹配
    0 => 引述 "零"
    _ => 引述 "非零"

设 结果1 = !加一(5)    // 6
设 结果2 = !描述(0)    // "零"
```

### 6.7.12 穷尽性检查

当模式匹配宏的所有分支无法覆盖所有可能的输入时，系统会发出警告：

```
// 缺少兜底分支 → 发出穷尽性警告
定义宏 严格(值) 匹配
    0 => 引述 "零"
    1 => 引述 "一"

// 建议添加通配符或变量绑定作为最后的分支
定义宏 完善(值) 匹配
    0 => 引述 "零"
    1 => 引述 "一"
    _ => 引述 "其他"     // 无警告
```

穷尽性检查仅在宏定义注册时执行，发出警告但不阻止注册。

### 6.7.13 展开策略

模式匹配宏采用**运行时展开**策略：在编译期将宏调用展开为等价的 `匹配` 语句 AST，由解释器在运行时执行模式匹配。这样设计的原因是：

- 宏调用参数经常是变量引用而非字面量（如 `!描述(x)`），编译期无法求值
- 复用现有 `匹配`/`情况` 的运行时匹配逻辑，无需重新实现
- 与道语言现有的模式匹配语义完全一致

### 6.7.14 错误处理

| 错误场景 | 行为 |
|----------|------|
| 调用未定义的模式匹配宏 | 抛出运行时错误："未找到宏定义 '名称'" |
| 参数数量不匹配 | 抛出运行时错误："宏 '名称' 期望N个参数，实际提供M个" |
| 无匹配模式 | 抛出运行时错误："模式匹配宏 '名称' 无法匹配输入值" |
| 宏重定义冲突 | 抛出宏展开错误："宏 '名称' 已在当前作用域中定义" |
| 空模式分支 | 抛出语法错误："模式匹配宏至少需要一个模式分支" |
