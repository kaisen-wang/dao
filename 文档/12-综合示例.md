# 12. 综合示例：多范式融合应用

为了具体展示"道"如何融合多种范式解决实际问题，我们构思一个场景：为一个简单的博客系统，根据文章标签和作者关系，推荐相关文章。

## 12.1 完整示例

```
// ============================================================
// 模块导入
// ============================================================
从 编码 导入 JSON
从 时间 导入 现在

// ============================================================
// 面向对象编程部分：定义核心数据结构
// ============================================================

/// 文章类型：表示博客中的一篇文章
类型 文章 实现 可序列化
    初始化(标题, 作者, 标签: 列表)
        本对象.标题 = 标题
        本对象.作者 = 作者
        本对象.标签 = 标签
        本对象.创建时间 = 现在()

    函数 转为文本()
        返回 `《{本对象.标题}》by {本对象.作者}`

/// 推荐结果：包含文章和相关度分数
类型 推荐项
    初始化(文章, 分数)
        本对象.文章 = 文章
        本对象.分数 = 分数

    函数 转为文本()
        返回 `{本对象.文章.转为文本()} (相关度: {本对象.分数})`

// ============================================================
// 逻辑/规则式编程部分：定义推荐规则知识库
// ============================================================

逻辑 推荐规则
    // 事实：作者之间的合作关系
    事实: 合作者("张三", "李四")
    事实: 合作者("李四", "王五")
    事实: 合作者("张三", "赵六")

    // 规则：合作关系是双向的
    规则: 相关作者(?A, ?B) 如果 合作者(?A, ?B)
    规则: 相关作者(?A, ?B) 如果 合作者(?B, ?A)

    // 规则：合作者的合作者也算间接相关（二度关系，分数较低）
    规则: 间接相关(?A, ?C) 如果
        合作者(?A, ?B) 并且 合作者(?B, ?C) 并且 ?A != ?C

// ============================================================
// 元编程/宏系统部分：定义实用宏
// ============================================================

/// 计时宏：自动测量代码块的执行耗时
定义宏 计时(名称, 代码块)
    返回 引述
        定义 __开始 = 系统.当前时间()
        定义 __结果 = { 注入 代码块 }
        定义 __耗时 = 系统.当前时间() - __开始
        打印(`[性能] {注入 名称} 耗时: {__耗时}毫秒`)
        __结果

// ============================================================
// 函数式编程部分：实现推荐算法
// ============================================================

/// 计算两篇文章之间的标签相似度
函数 标签相似度(文章甲, 文章乙)
    定义 共同标签 = 交集(
        集合(文章甲.标签),
        集合(文章乙.标签)
    )
    返回 长度(共同标签)

/// 生成推荐列表
/// 参数 当前文章：要为其生成推荐的文章
/// 参数 所有文章：候选文章库
/// 返回：按相关度降序排列的推荐列表
函数 生成推荐(当前文章: 文章, 所有文章: 列表) -> 列表
    返回 所有文章
        // 1. 排除当前文章
        |> 筛选(函数(a) => a.标题 != 当前文章.标题)
        // 2. 计算每篇文章的相关度分数
        |> 映射(函数(候选文章)
            定义 分数 = 0

            // 标签匹配度（每个共同标签 +10 分）
            分数 = 分数 + 标签相似度(当前文章, 候选文章) * 10

            // 作者直接相关度（+50 分）
            定义 直接相关 = 查询(推荐规则, 相关作者(当前文章.作者, 候选文章.作者))
            如果 不是 空列表(直接相关)
                分数 = 分数 + 50

            // 作者间接相关度（+20 分）
            定义 间接相关 = 查询(推荐规则, 间接相关(当前文章.作者, 候选文章.作者))
            如果 不是 空列表(间接相关) 并且 空列表(直接相关)
                分数 = 分数 + 20

            返回 推荐项(候选文章, 分数)
        )
        // 3. 过滤掉分数为 0 的文章
        |> 筛选(函数(项) => 项.分数 > 0)
        // 4. 按分数降序排序
        |> 排序(键=函数(项) => 项.分数, 降序=真)

// ============================================================
// 主程序：组合所有部分
// ============================================================

// 创建文章数据
定义 文章库 = [
    文章("道语言设计", "张三", ["编程", "语言设计", "中文"]),
    文章("虚拟机技术深入", "李四", ["编程", "性能优化"]),
    文章("AI 新进展", "王五", ["AI", "科技"]),
    文章("语言设计的未来", "赵六", ["语言设计", "未来趋势"]),
    文章("编程入门指南", "孙七", ["编程", "教育"]),
]

// 使用计时宏来测量推荐算法性能
定义 目标文章 = 文章库.0  // "道语言设计" by 张三

!计时("推荐生成", {
    定义 推荐列表 = 生成推荐(目标文章, 文章库)

    打印(`\n为 {目标文章.转为文本()} 推荐的文章：`)
    打印("─" * 40)

    遍历 索引, 推荐 在 枚举(推荐列表)
        打印(`  {索引 + 1}. {推荐.转为文本()}`)

    如果 空列表(推荐列表)
        打印("  暂无推荐")
})
```

## 12.2 预期输出

```
为 《道语言设计》by 张三 推荐的文章：
────────────────────────────────────────
  1. 《虚拟机技术深入》by 李四 (相关度: 60)
  2. 《语言设计的未来》by 赵六 (相关度: 60)
  3. 《AI 新进展》by 王五 (相关度: 20)
  4. 《编程入门指南》by 孙七 (相关度: 10)
[性能] 推荐生成 耗时: 3毫秒
```

**分数计算说明：**
- **虚拟机技术深入**（李四）：共同标签"编程"(+10) + 张三与李四直接合作(+50) = 60
- **语言设计的未来**（赵六）：共同标签"语言设计"(+10) + 张三与赵六直接合作(+50) = 60
- **AI 新进展**（王五）：无共同标签 + 张三与王五间接相关(通过李四)(+20) = 20
- **编程入门指南**（孙七）：共同标签"编程"(+10) + 无作者关系 = 10

## 12.3 范式协作分析

这个例子清晰地展示了四种范式如何各司其职、无缝协作：

| 范式 | 承担的角色 | 具体体现 |
|------|-----------|---------|
| **面向对象** | 数据建模 | `文章`、`推荐项` 类型封装数据和行为 |
| **逻辑编程** | 声明式知识推理 | `推荐规则` 知识库定义作者关系，自动推理直接/间接相关 |
| **元编程** | 横切关注点 | `计时` 宏自动注入性能测量代码，无侵入业务逻辑 |
| **函数式编程** | 数据转换管道 | 管道运算符 `\|>` 串联 映射→筛选→排序，清晰声明式 |

四种范式不是简单堆砌，而是相互增强：
- OOP 的对象可以传入 FP 的管道
- FP 的查询函数可以调用逻辑引擎
- 宏可以包裹任何范式的代码块
- 逻辑查询结果可以流入 FP 的管道处理
