## 第四批：运算符补全（优先级：中低）

### 任务 4.1：整除运算符 `//`

- **状态**：❌ 未实现
- **设计规格**：`10 // 3` → `3`；行首 `//` 仍为注释
- **涉及文件**：
  - `lexer/core.py` — 修改注释识别逻辑，仅在行首/空白后 `//` 为注释，表达式中 `//` 为整除
  - `lexer/readers.py` — 新增整除 token 读取
  - `parser/expressions.py` — 新增整除运算解析
  - `interpreter/expressions.py` — 新增整除求值
- **关键难点**：**这是最棘手的设计问题**。词法分析器在 tokenize 阶段无法区分上下文。可选方案：
  - **方案 A**：词法分析器始终将 `//` 识别为整除 token，注释改用 `注：` 或 `#`，但会破坏现有代码
  - **方案 B**：词法分析器根据前一个 token 类型判断——若前一个 token 是数值/标识符/右括号等，则 `//` 为整除；否则为注释
  - **方案 C**：推迟到语法分析阶段处理，词法分析器输出 `双斜杠` token，由 parser 根据上下文决定
  - **推荐方案 B**，改动最小且兼容现有代码
- **测试**：至少 5 个测试用例（含边界情况）
- **依赖**：无

### 任务 4.2：逻辑运算符 `&&`/`||`/`!`

- **状态**：❌ 未实现
- **设计规格**：`&&` 等价 `并且`，`||` 等价 `或者`，`!` 等价 `不是`
- **涉及文件**：
  - `tokens.py` — 新增 `TokenType.逻辑与`、`TokenType.逻辑或`
  - `lexer/readers.py` — 识别 `&&`、`||` 符号
  - `parser/expressions.py` — 将 `&&`/`||`/`!` 映射到与 `并且`/`或者`/`不是` 相同的 AST 节点
  - `interpreter/expressions.py` — 无需修改（复用现有逻辑）
- **关键难点**：`!` 当前被用作宏调用前缀，需确保不冲突
- **测试**：至少 3 个测试用例
- **依赖**：无
