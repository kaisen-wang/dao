## 2. Channel 异步方法非原生实现

**现状**：`send_async()`/`receive_async()` 使用 `run_in_executor` 包装同步方法

**问题**：
- 不是真正的 asyncio 原生异步，存在线程切换开销
- 在已有事件循环中可能导致死锁

**任务**：
- [ ] 基于 `asyncio.Queue` 实现原生异步通道
- [ ] `send_async()`/`receive_async()` 改为 `async def` 原生协程
- [ ] 保持同步 `send()`/`receive()` 接口不变（向后兼容）

---

## 3. 缺少高级同步原语

**现状**：仅有 `Mutex`（互斥锁）、`AtomicInt`、`AtomicBool`

**缺失**：
- [ ] `Semaphore`（信号量）— 限制并发访问数量
- [ ] `WaitGroup`/`Barrier`（等待组/屏障）— 多协程同步点
- [ ] `RLock`（可重入锁）— 同一协程可多次获取
- [ ] `Condition`（条件变量）— 等待/通知模式
- [ ] `Once`（单次执行）— 初始化保护

---

## 4. 缺少真正的协程调度器

**现状**：并行块 (`ParallelStmt`) 使用 `ThreadPoolExecutor` 执行

**问题**：
- 线程开销大，不适合大量轻量级协程
- 无法实现真正的协程调度（如工作窃取、优先级调度）

**任务**：
- [ ] 实现基于 `asyncio` 的轻量级协程调度器
- [ ] 支持协程创建、挂起、恢复、取消
- [ ] 支持协程间通信（通过通道）
- [ ] 实现协程池（限制并发协程数量）

---
