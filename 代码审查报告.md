# "道"编程语言代码审查报告

## 项目概览

- **项目**：道（Dao）— 基于简体中文的多范式编程语言
- **实现语言**：Python 3.10+
- **代码规模**：核心源码 ~17,555 行 / 测试 ~12,553 行
- **架构**：词法分析 → 语法分析 → AST → 树遍历解释器 / 字节码编译 → VM

---

## 一、严重问题 (Critical)

### 1. `len()` 函数误用 — 运行时崩溃

**文件**: `源码/dao/interpreter/expressions.py:520, 535, 542`

```python
# 错误：len(obj, 0, 0, self.source) — len() 只接受1个参数
raise 索引错误(
    f"列表索引 {idx} 超出范围 (长度 {len(obj, 0, 0, self.source)})",
)
```

`len()` 被错误地传入了4个参数，这会在运行时直接抛出 `TypeError`，导致索引越界检查本身崩溃。同样的问题出现在 `eval_index_access` 方法的列表、字符串和类型错误分支中。

### 2. `type()` 函数误用 — 运行时崩溃

**文件**: `源码/dao/interpreter/statements.py:299`

```python
f"无法给类型 '{type(obj, 0, 0, self.source).__name__}' 设置索引",
```

`type()` 只接受1个参数，这里错误地传入了4个参数。

### 3. 三引号字符串未闭合检测失效

**文件**: `源码/dao/lexer/readers.py:154-156`

```python
if self.current_char is None and not (len(result) >= 0):
    pass
```

条件 `not (len(result) >= 0)` 永远为 `False`（因为 `len(result) >= 0` 恒为 `True`），所以这个 `if` 块永远不会执行。这意味着未闭合的三引号字符串不会报错，而是静默返回可能不完整的内容。

### 4. `DaoInstance.get_field` 无法区分 `None` 值和字段不存在

**文件**: `源码/dao/builtins/oop_types.py:112-119`

```python
def get_field(self, name: str):
    if name in self.fields:
        return self.fields[name]
    method = self.klass.find_method(name)
    if method is not None:
        return BoundMethod(self, method)
    return None
```

当字段值为 `None` 时，`get_field` 返回 `None`，调用方无法区分"字段值为 None"和"字段不存在"。在 `expressions.py:253-261` 中：

```python
result = obj.get_field(expr.member)
if result is not None:
    return result
raise 名称错误(...)
```

如果字段值确实为 `None`，会错误地抛出 `名称错误`。

## 二、重要问题 (Major)

### 5. `MacroRegistry` 单例模式导致状态泄漏

**文件**: `源码/dao/macros/registry.py:48-55`

```python
class MacroRegistry:
    _instance = None
    def __new__(cls):
        if cls._instance is None:
            cls._instance = super().__new__(cls)
            cls._instance._initialize()
        return cls._instance
```

`MacroRegistry` 使用单例模式，但 `Interpreter.execute()` 每次执行时都调用 `registry.clear()`（`源码/dao/interpreter/core.py:87-88`）。这意味着：
- 多个 Interpreter 实例共享同一个宏注册表
- 一个 Interpreter 执行 `clear()` 会影响其他 Interpreter
- 在并发场景下可能导致竞态条件

### 6. `ErrorInstanceWrapper` 重复定义

**文件**: `源码/dao/interpreter/core.py:179-196` 和 `源码/dao/interpreter/expressions.py:299-312`

`ErrorInstanceWrapper` 类在两个文件中分别定义了两次，且实现略有不同。这是代码重复，增加了维护成本和不一致风险。

### 7. `_exec_block` 方法重复定义

**文件**: `源码/dao/interpreter/core.py:380-385` 和 `源码/dao/interpreter/expressions.py:938-948`

`_exec_block` 在 `Interpreter`（通过 `StatementExecutor` 混入）和 `ExpressionEvaluator` 中各定义了一次，且实现不同。`ExpressionEvaluator` 版本额外处理了 `ReturnStmt` 和 `expression` 属性，而 `StatementExecutor` 版本更简单。由于 MRO（方法解析顺序），`ExpressionEvaluator` 的版本会覆盖 `StatementExecutor` 的版本，可能导致非预期行为。

### 8. `exec_logic_block` 中大量重复代码

**文件**: `源码/dao/interpreter/statements.py:1044-1119`

处理 `FunctionCall` 和 `LogicFact` 两种 body 表达式类型的代码块几乎完全相同（约 70 行重复），仅 `predicate` 的获取方式不同。应提取为公共方法。

### 9. VM 中 `LOAD_LOCAL` 对未定义变量返回 `None` 而非报错

**文件**: `源码/dao/vm/core.py:48-52`

```python
elif op == OpCode.LOAD_LOCAL:
    name = frame.code.local_names[operand]
    if name in frame.locals:
        frame.push(frame.locals[name])
    else:
        frame.push(None)
```

树遍历解释器对未定义变量会抛出 `名称错误`，但 VM 静默返回 `None`。两种执行模式行为不一致，可能导致在 VM 模式下隐藏 bug。

### 10. 字节码编译器不支持 Break/Continue

**文件**: `源码/dao/bytecode/compiler.py`

`_compile_statement` 方法没有处理 `BreakStmt` 和 `ContinueStmt`，在 VM 模式下使用 break/continue 会静默忽略，行为与树遍历解释器不一致。

## 三、一般问题 (Minor)

### 11. `//` 注释与整除的歧义判断逻辑复杂且脆弱

**文件**: `源码/dao/lexer/core.py:179-223`

通过前一个 token 类型和后续字符来区分 `//` 是注释还是整除运算符，逻辑复杂且容易出错。建议使用不同的注释语法（如 `#` 或保留 `注：`）来避免歧义。

### 12. `_read_identifier` 中前瞻回退机制脆弱

**文件**: `源码/dao/lexer/readers.py:317-377`

`异步` 和 `否则` 关键字的前瞻检测通过保存/恢复 `pos`、`line`、`column` 实现。如果前瞻过程中有副作用（如修改 `_bracket_depth`），回退可能不完整。

### 13. `DaoGenerator` 直接操作 `env.values` 绕过封装

**文件**: `源码/dao/builtins/oop_types.py:212, 232`

```python
env.values[stmt.variable] = item
env.values[stmt.variable] = current
```

直接修改 `env.values` 字典绕过了 `Environment.define()` 的常量检查和重定义检查。

### 14. `eval_member_access` 方法过长

**文件**: `源码/dao/interpreter/expressions.py:214-414`

`eval_member_access` 方法约 200 行，处理了 `DaoInstance`、`DaoError`、`SuperProxy`、`DaoEnum`、`DaoClass`、`dict`、`str`、`list`、`Channel`、`Mutex`、`AtomicInt`、`AtomicBool` 等多种类型。建议按类型拆分为独立方法。

### 15. `_get_str_method` / `_get_list_method` 等方法每次调用都重建字典

**文件**: `源码/dao/interpreter/expressions.py:416-509`

每次访问字符串/列表方法时都重新创建 `methods` 字典和 `BuiltinFunction` 对象，在频繁调用的热路径上会产生不必要的内存分配。建议缓存或使用类级别的映射。

### 16. `solver.py` 中 `query()` 函数使用绝对导入 `from dao.xxx`

**文件**: `源码/dao/logic/solver.py:237, 261, 269, 287, 291`

```python
from dao.ast_nodes import FunctionCall
from dao.logic.core import LogicVariable
```

其他模块都使用相对导入（`from ..xxx`），但 `query()` 函数使用了绝对导入，在包作为库使用时可能失败。

### 17. `Semaphore.value` 访问私有属性

**文件**: `源码/dao/concurrency/sync.py:104`

```python
return self._semaphore._value
```

直接访问 `threading.Semaphore` 的私有属性 `_value`，这在 Python 版本升级时可能失效。

### 18. `exec_class_decl` 中错误类和普通类的代码重复

**文件**: `源码/dao/interpreter/statements.py:682-711`

`is_error_class` 和 `else` 分支的 `DaoClass` 构造和 `env.define` 调用完全相同，仅 `parent` 参数不同。可以简化。

### 19. `_add_constant` 使用 `is` 而非 `==` 比较常量

**文件**: `源码/dao/bytecode/compiler.py:296`

```python
if c is value:
    return i
```

使用 `is` 比较意味着值相同但对象不同的常量不会被去重（如两个值为 `1` 的 `NumberLiteral`），降低了常量池的效率。

### 20. `eval_expression` 中使用 `type().__name__` 字符串比较而非 `isinstance`

**文件**: `源码/dao/interpreter/expressions.py:33`

```python
if type(expr).__name__ == "LogicVariable":
```

应使用 `isinstance` 检查，字符串比较容易因重命名而失效。

## 四、设计建议

### 21. 考虑使用访问者模式替代 Mixin + match

当前解释器使用 Mixin 模式 + `match/case` 分派，导致：
- `eval_member_access` 等方法过长
- 新增类型需要修改多处代码
- `_exec_block` 等方法在多个 Mixin 中重复定义

建议引入访问者模式（Visitor Pattern），将每种 AST 节点的处理逻辑集中管理。

### 22. 统一树遍历解释器和 VM 的语义

当前两种执行模式存在多处行为不一致（未定义变量处理、break/continue 支持等），建议建立统一的语义测试套件，确保两种模式行为一致。

### 23. `ErrorInstanceWrapper` 应统一为正式的类型适配器

当前 `ErrorInstanceWrapper` 作为内部类在两处临时定义，建议提取为 `DaoError` 的正式适配器，实现 `DaoInstance` 接口。
